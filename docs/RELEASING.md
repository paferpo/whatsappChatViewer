# Releasing

How to cut a new desktop release (macOS `.dmg` + Windows `.msi`/`.exe`).

## Automated: `npm run release`

[scripts/release.mjs](../scripts/release.mjs) does steps 1–3 below in one go:
bumps every version file listed in step 1, commits, tags, and pushes.

```
npm run release -- "short summary of what's in this release"
```

- Defaults to a patch bump; pass `--minor` or `--major` to bump differently,
  or `--set=X.Y.Z` for an explicit version.
- Shows the pending `git status` and asks for confirmation before committing
  and pushing (skip the prompt with `--yes`).
- After it finishes, continue from step 4 (watch the workflow) below.

## Manual steps

## 1. Bump the version

All of these must be bumped together to the same `X.Y.Z` — they're independent
files (npm and Cargo don't share a version), and a mismatch is easy to miss:

| File | Field |
|---|---|
| [package.json](../package.json) | `version` |
| [package-lock.json](../package-lock.json) | top-level `version` **and** `packages.".".version` |
| [src-tauri/tauri.conf.json](../src-tauri/tauri.conf.json) | `version` |
| [src-tauri/Cargo.toml](../src-tauri/Cargo.toml) | `[package].version` |
| [src-tauri/Cargo.lock](../src-tauri/Cargo.lock) | the `[[package]] name = "app"` entry's `version` |

`package-lock.json` and `Cargo.lock` are normally regenerated by their
respective package managers (`npm install --package-lock-only`, `cargo build`),
but editing the version fields by hand is fine and avoids pulling in unrelated
dependency churn.

## 2. Commit

```
git add -A
git commit -m "Bump to X.Y.Z: <short summary of what's in this release>"
git push origin master
```

## 3. Tag and push

```
git tag vX.Y.Z
git push origin vX.Y.Z
```

The tag push (matching `v*`) triggers
[`.github/workflows/release.yml`](../.github/workflows/release.yml), which:
- builds the app for macOS (aarch64) and Windows,
- runs `npm run build` then `tauri build` via `tauri-action`,
- uploads the resulting bundles to a **draft** GitHub Release named after the
  tag.

## 4. Watch the workflow

```
gh run list --workflow=release.yml --limit 3
```

Wait for `completed success`. If it shows `cancelled` or `failure`, the
release did not ship — check the run logs (`gh run view <run-id> --log-failed`)
before assuming the tag is live.

## 5. Publish

Releases are created as **drafts**. Open the repo's Releases page, check the
attached artifacts, then publish manually. Installs are unsigned:
- **macOS:** right-click the `.app` → Open (or
  `xattr -dr com.apple.quarantine /path/to/WhatsappChatViewer.app`).
- **Windows:** "More info" → "Run anyway" past SmartScreen.

## Local build (no release)

`npm run tauri:build` builds locally and produces installers under
`src-tauri/target/release/bundle/`. Useful for testing before tagging.

### Known local gotcha: stale `dmg.*` mounts

If `tauri:build` fails during the macOS DMG step (`bundle_dmg.sh` failing to
run), a previous failed build may have left disk images mounted. Check with:

```
hdiutil info | grep -E "^/dev/disk|image-path"
```

Detach any stale ones (e.g. `/Volumes/dmg.xxxxx` or leftover images from a
since-renamed project path) with `hdiutil detach /dev/diskN -force`, then
retry the build.

If the project directory itself was ever renamed, also clear the Cargo build
cache (`rm -rf src-tauri/target` — gitignored, fully regenerable) since it can
hold fingerprints pointing at the old absolute path.
